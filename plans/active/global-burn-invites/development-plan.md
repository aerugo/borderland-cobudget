# Global Burn Invites - Development Plan

**Status**: Draft
**Created**: 2026-04-18
**Branch**: `borderland` (current) — will likely cut a dedicated feature
branch before implementation
**Spec**: [spec.md](spec.md)

## Summary

Add a new "Invite members" round settings tab whose first section lets round
admins configure a Global Burn instance (URL + event ID + API key), verify
the connection, preview how many members would be invited, and fire the
existing bulk-invite flow against those members.

## Current State Analysis

### What exists today

- **Bulk invite backend**: `inviteRoundMembersHelper` in
  [ui/server/graphql/resolvers/helpers/inviteRoundMemberHelpers.js](../../../ui/server/graphql/resolvers/helpers/inviteRoundMemberHelpers.js)
  already handles comma-separated email strings up to 10,000 entries. It
  de-duplicates against existing `RoundMember` records, creates `User` +
  `Account` + `RoundMember` rows, and invokes a caller-supplied
  `onMembersToInvite` callback for the email side.
- **Bulk invite email**: `emailService.bulkInviteMembers` in
  [ui/server/services/EmailService/email.service.ts](../../../ui/server/services/EmailService/email.service.ts)
  is called by the existing `inviteRoundMembers` mutation. Behind it,
  [ui/server/send-email.ts](../../../ui/server/send-email.ts) `sendBatch()`
  already splits into parallel Postmark batches of 500, which means "send
  to thousands" is covered at the transport layer.
- **Round settings tabs**: Defined as an array in
  [ui/components/RoundSettings/index.tsx:53-119](../../../ui/components/RoundSettings/index.tsx#L53-L119).
  Each entry is `{ slug, name, component }`; adding a tab is a single
  array insert. Admin gating lives in
  [ui/pages/[group]/[round]/settings/[[...settingsTab]].tsx](../../../ui/pages/%5Bgroup%5D/%5Bround%5D/settings/%5B%5B...settingsTab%5D%5D.tsx).
- **External-secret precedent**: `ocToken` on Round is stored as plain
  string but exposed via GraphQL only as `ocTokenStatus: TOKEN_STATUS`
  (`EMPTY` | `PROVIDED`). See
  [ui/server/graphql/resolvers/types/Round.ts:373-377](../../../ui/server/graphql/resolvers/types/Round.ts#L373-L377).
  `ocVerified` boolean is the precedent for tracking "verified connection"
  state. We mirror both patterns.
- **Existing invite UI**: `InviteMembersModal` at
  [ui/components/InviteMembersModal.tsx](../../../ui/components/InviteMembersModal.tsx)
  is reached via a button on the Round Members page. We are **not**
  touching this — the new tab is a separate entry point.

### Problem we're solving

Admins currently have to leave Cobudget, go to the Global Burn admin UI,
export emails, and paste them into `InviteMembersModal`. We replace that
round-trip with a server-side fetch.

## Solution Design

### Data flow

```
Admin → RoundSettings/InviteMembers/GlobalBurnSection
  │
  ├── editGlobalBurnSettings mutation
  │     → Round.globalBurn{InstanceUrl,EventId,ApiKey} written
  │     → globalBurnVerified reset to false
  │
  ├── testGlobalBurnConnection mutation
  │     → GlobalBurnService.fetchMembers()
  │     → classify result (OK | INVALID_KEY | EVENT_NOT_FOUND | UNREACHABLE)
  │     → on OK: set globalBurnVerified=true, return member count
  │     → on failure: set globalBurnVerified=false, return error enum
  │
  └── syncGlobalBurnMembers mutation (called twice)
        ├── dryRun=true → return { totalInEvent, alreadyMembers, toInvite }
        └── dryRun=false → call inviteRoundMembersHelper with the fresh email
                           list + bulkInviteMembers callback → return counts
```

### Key Design Decisions

1. **Reuse `inviteRoundMembersHelper` verbatim**. Don't write a parallel
   invite path. The helper already de-duplicates against existing
   `RoundMember` rows, enforces the 10k cap, and handles the hard parts
   (account creation, batch writes). Calling it with a joined CSV string
   is the same interface the existing `inviteRoundMembers` mutation uses.

2. **API key never returns to the client**. Mirror `ocToken`: store the raw
   string in DB, expose only `globalBurnApiKeyStatus: TOKEN_STATUS` on
   the Round type. The `editGlobalBurnSettings` mutation accepts
   `apiKey: String` with tri-state semantics (null = unchanged, empty =
   clear, non-empty = set). The UI password field submits `null` when the
   admin hasn't touched it.

3. **Server-side fetch, not client-side**. The API key lives only on the
   server. The two-step sync (`dryRun=true` then `dryRun=false`) both hit
   the Global Burn API anew — a few extra seconds of latency is fine, and
   it avoids caching potentially-sensitive email lists in Node memory
   between requests.

4. **Error mapping lives in the service, not the resolver**. `GlobalBurnService.fetchMembers` returns a discriminated union
   `{ status: "OK", emails } | { status: "INVALID_KEY" } | ...`. Resolvers
   translate that into GraphQL response shape. Keeps HTTP concerns out of
   the GraphQL layer.

5. **No new background/queue system**. `bulkInviteMembers` is already
   fire-and-forget via Postmark's batch API. For a 3,000-member sync, the
   mutation waits on DB writes (~seconds) but not on email delivery.
   If a future event has >10k members we'll revisit — explicitly out of
   scope for v1.

6. **SSRF guard before the fetch (prod only)**. Reject instance URLs
   whose hostnames match localhost, RFC1918 private ranges, or link-local
   (`169.254.*`), and require `https://`. Admins entering arbitrary URLs
   is a new surface; block it early with a simple hostname string check
   (no DNS resolution — admins get a clear error, and DNS resolution
   opens a check-vs-fetch race). Bypass the guard when
   `process.env.NODE_ENV === "development"` so a developer can point the
   feature at `http://localhost:<port>` running a local mock Burn server.
   Unit tests must set `NODE_ENV=production` explicitly to exercise the
   blocking logic (default Vitest env would silently bypass it).

## Phase 1: Data Model & Settings Mutation

**Goal**: Admin can save Global Burn config (URL, event ID, API key) on a
round via GraphQL. API key is never returned. No UI yet.
**Detailed Plan**: [phases/phase-1.md](phases/phase-1.md)

### Deliverables

1. Prisma migration adding four fields to `Round`.
2. `editGlobalBurnSettings` mutation in
   [round.ts](../../../ui/server/graphql/resolvers/mutations/round.ts).
3. `globalBurnApiKeyStatus` resolver in
   [types/Round.ts](../../../ui/server/graphql/resolvers/types/Round.ts).
4. Schema updates in
   [schema/index.js](../../../ui/server/graphql/schema/index.js).

### Implementation Approach

1. Add fields to `schema.prisma`; run `yarn migrate`.
2. Extend the `Round` SDL type with the three non-secret fields + the
   status enum field.
3. Add the mutation to the SDL + resolver map.
4. Implement the mutation with the tri-state `apiKey` semantics and the
   `globalBurnVerified` reset behavior.
5. Add the `globalBurnApiKeyStatus` type resolver (gated to admins,
   same as `ocTokenStatus`).

### Success Criteria

- [ ] `yarn typecheck` passes.
- [ ] Playground mutation saves + clears the key as expected.
- [ ] Querying `globalBurnApiKeyStatus` returns `PROVIDED` after save,
      `EMPTY` after clear; never returns the raw string.
- [ ] Editing any config field resets `globalBurnVerified` to `false`.

## Phase 2: Connection Test + Sync Backend

**Goal**: Backend can test the connection (returns typed error info) and
can run a dry-run or real sync against the existing invite helper.
**Detailed Plan**: phases/phase-2.md (to be written when Phase 1 lands)

### Deliverables

1. `ui/server/services/GlobalBurnService.ts` with `fetchMembers(round)`
   returning a discriminated union.
2. `testGlobalBurnConnection` mutation.
3. `syncGlobalBurnMembers(dryRun)` mutation.
4. New GraphQL result types + enum.

### Implementation Approach

1. Write `GlobalBurnService.fetchMembers()`:
    - Build URL from round config.
    - Use `fetch()` with `AbortController` + ~10s timeout.
    - SSRF hostname check before dispatch.
    - Map response codes: 200→OK, 403→INVALID_KEY, 400→EVENT_NOT_FOUND,
      everything else (DNS, TCP, 5xx, non-JSON) → UNREACHABLE.
    - Validate response is `string[]` of lowercase-trimmed emails.
2. Implement `testGlobalBurnConnection`: call service, update
   `globalBurnVerified`, return `{ status, memberCount, error }`.
3. Implement `syncGlobalBurnMembers`:
    - Call service. If not OK, throw with the same error enum.
    - Fetch current `RoundMember` emails to compute `alreadyMembers` /
      `toInvite` counts.
    - If `dryRun`, return counts without invoking the helper.
    - Otherwise call `inviteRoundMembersHelper` with
      `toInvite.join(",")` and the standard `bulkInviteMembers` callback.
4. Wire mutations into the resolver map.

### Success Criteria

- [ ] `yarn typecheck` passes.
- [ ] Mutation against a known-good instance returns `status: OK` and
      a plausible member count.
- [ ] Feeding a wrong API key returns `INVALID_KEY` and leaves
      `globalBurnVerified=false`.
- [ ] Feeding a bogus event ID returns `EVENT_NOT_FOUND`.
- [ ] Feeding an unreachable host returns `UNREACHABLE`.
- [ ] `dryRun=true` never writes to DB (verify via `yarn prisma studio`).
- [ ] `dryRun=false` creates `RoundMember` rows and enqueues emails.

## Phase 3: Settings UI

**Goal**: Round admins can do the whole flow from the browser without
leaving settings.
**Detailed Plan**: phases/phase-3.md (to be written after Phase 2)

### Deliverables

1. New `"invite-members"` tab in `RoundSettings/index.tsx`.
2. `InviteMembers/index.tsx` tab root.
3. `InviteMembers/GlobalBurnSection.tsx` with:
    - Config form (URL + Event ID + password-style API key).
    - Save button.
    - Test Connection button with typed error messaging.
    - Conditional "Fetch members" button (shown only when verified).
    - Counts display + "Send invites" button with loading state.

### Implementation Approach

1. Add tab entry (one array push) + import.
2. Scaffold `InviteMembers/index.tsx` as a wrapper that renders
   `GlobalBurnSection` (leaves room for future sections).
3. Build `GlobalBurnSection` using the same `react-hook-form` + Urql
   patterns as `GeneralSettings.js` and `SetOCToken.tsx`.
4. Hook up error messages per the four outcomes in AC4.
5. Two-step sync UX: "Fetch members" runs dry-run, shows counts, reveals
   "Send invites" button. "Send invites" runs the real mutation and
   surfaces success via toast.

### Success Criteria

- [ ] Non-admins don't see the tab (covered by existing page gate).
- [ ] Manually testing the four connection outcomes surfaces the right
      messages in the UI.
- [ ] Sending invites for ~100 fake members (local DB + mock server)
      creates RoundMembers and enqueues emails in dev mode.
- [ ] Password field is blank after reload; `globalBurnApiKeyStatus`
      label shows `PROVIDED` or `EMPTY` next to it.

## Phase 4: Polish + Verification

**Goal**: Ship-ready. Cache consistent, large-batch tested, no regressions.

### Deliverables

1. Urql cache update rules in `ui/graphql/client.ts` matching the
   existing `inviteRoundMembers` invalidation.
2. Manual test with a realistic (mocked) response of ~3,000 emails.
3. Documentation comment in `GlobalBurnService.ts` describing the API
   contract we depend on (so the next maintainer can spot a breaking
   change).

### Success Criteria

- [ ] Round Members list updates in the UI after a successful sync
      without a manual refresh.
- [ ] 3,000-email dry-run returns counts in under ~5s; real sync
      completes DB writes in under ~60s.
- [ ] No TypeScript or Vitest regressions.
- [ ] `yarn test:run` green.

## Testing Strategy

### Unit Tests (Vitest)

- `ui/__tests__/globalBurnService.test.ts`:
    - Mocks `fetch` and asserts the discriminated union for each
      HTTP status + network failure case.
    - Asserts SSRF hostname check rejects `localhost`, `127.0.0.1`,
      `10.x`, `192.168.x`, `169.254.x`.
    - Asserts response validation rejects non-array / non-string bodies.

### Manual / E2E

- Mock a Global Burn server locally (a tiny Express app returning a
  JSON array of emails) to exercise the full flow.
- No Cypress spec in v1 — the flow involves an external service and
  adding a mock server to the Cypress setup is disproportionate for
  the first release. Revisit after it's been in production.
